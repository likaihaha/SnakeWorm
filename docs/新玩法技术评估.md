# SnakeWorm 新玩法技术评估：Z字形关卡结构

> 创建时间：2026-05-03
> 基于 v1.68 架构评估

---

## 一、概述

**目标**：将 SnakeWorm 从开放世界改为 Z 字形关卡结构——横向 5 关卡、纵向 5 层，出生点在左下第一个 800×600 空间，满足条件后解锁下一关。

---

## 二、正面评价

### 结构优势
- **路径清晰**：Z字形给玩家明确的"下一步去哪"的目标感，不会迷路
- **空间利用**：4000×3000 的地图恰好 = 5列×5行 × 800×600
- **渐进式体验**：可以设计成"从底层一路打上去"的成就感
- **叙事潜力**：很适合设计成"虫虫从地底爬向地面/天空"的故事线

### 游戏设计
- **条件解锁**：给玩家阶段性目标（比如吃够 10 个宝珠、达到某长度）
- **难度曲线**：每层可以引入新机制或更强敌人
- **重玩价值**：可以设计隐藏区域或彩蛋关卡

---

## 三、反面评估/挑战

### 技术实现问题

| 问题 | 说明 |
|------|------|
| **架构冲突** | 当前是开放世界+相机跟随，改为关卡制需要大量重构（边界检测、区域切换、进度存储） |
| **障碍系统** | 需要新增前景障碍物渲染层，当前只有背景瓦片和实体 |
| **条件触发** | 需要一套"满足条件→解锁下一关"的事件系统 |
| **状态管理** | 需要保存玩家进度（已解锁关卡、当前关卡）到 LocalStorage |

### 游戏体验风险

- **线性限制**：失去开放世界的自由探索感，可能让部分玩家觉得"被限制"
- **失败惩罚**：如果死亡要重头开始，体验会很差；需要设计检查点/重生机制
- **重复感**：如果每层玩法差异不大，Z字形路径会显得单调
- **视野问题**：800×600 的关卡空间对贪吃蛇来说可能偏小，转弯时容易撞墙

### 平衡难度

- 条件门槛太高 → 玩家卡关流失
- 条件门槛太低 → 关卡切换太快，缺乏成就感
- 障碍物太难 → 挫败感
- 障碍物太简单 → 无挑战性

## 四、技术深度分析

### 当前架构核心参数

| 参数 | 当前值 | Z字形适配 |
|------|--------|-----------|
| 地图尺寸 | 4000×3000 | 恰好 = 5列×5行 × 800×600 ✅ |
| 视口尺寸 | 800×600 | 恰好 = 1个关卡空间 ✅ |
| 出生点 | (400, 2800) | 左下角第1格 ✅ |

## 五、需要新增/修改的系统

### 1. 区域管理系统 (ZoneManager) — 全新

```javascript
class ZoneManager {
    constructor() {
        this.zones = [];           // 25个Zone对象
        this.currentZoneId = 0;    // 当前所在关卡
        this.unlockedZones = new Set([0]); // 已解锁关卡
        this.progress = {};        // 各关卡的通关条件进度
    }

    // 根据世界坐标判断所在关卡
    getZoneAt(worldX, worldY) {
        const col = Math.floor(worldX / 800);
        const row = Math.floor(worldY / 600);
        return this.zones[row * 5 + col];
    }

    // 检查玩家是否跨区了
    checkZoneTransition(playerX, playerY) { ... }

    // 检查通关条件
    checkUnlockCondition(zoneId) { ... }
}
```

### 2. 障碍/门系统 (Barrier) — 全新

```javascript
class Barrier {
    constructor(x, y, width, height, condition) {
        this.bounds = { x, y, width, height };
        this.condition = condition;  // { type: 'score', target: 100 }
        this.active = true;          // 是否仍然阻挡
        this.opening = false;        // 正在打开动画
    }

    // 检查实体是否碰到障碍
    collidesWith(entityX, entityY, radius) { ... }

    // 检查是否满足解锁条件
    tryUnlock(playerStats) { ... }
}
```

**障碍物放置策略**：
- **横向关卡间**：在两格交界处放垂直墙壁（带缺口/门）
- **纵向转角**：在上下层交界处放水平墙壁
- 每个障碍有**通过条件**（吃到N个宝珠/达到某长度/消灭N个敌人/活过N秒）

### 3. 关卡内实体管理 — 修改现有

```javascript
// 当前问题：Food.random() 在出生点周围半径1200内随机生成
// 玩家走到关卡5时，宝珠可能在关卡1生成

// 改造方案：
class Zone {
    foods = [];        // 该区域内的宝珠
    enemies = [];      // 该区域内的敌人
    spawnPoints = [];  // 该区域的生成点

    update(dt, isActive) {
        // 只有当前区域+相邻区域才做完整update
        // 远处区域：暂停实体运动、停止生成
        if (isActive) {
            this.spawnFood(dt);
            this.spawnEnemies(dt);
        }
    }
}
```

### 4. 相机边界处理 — 小改

- **过渡动画**：玩家从关卡1走到关卡2时，相机平滑横移 800px
- **锁定视角**：在关卡切换时可短暂锁定相机，做过渡效果
- 当前 Camera.follow() 已有 lerp，天然支持

### 5. 进度存储 — 新增

```javascript
class ProgressManager {
    static STORAGE_KEY = 'snakeworm_progress';

    save(progress) {
        localStorage.setItem(this.STORAGE_KEY, JSON.stringify({
            unlockedZones: [...progress.unlockedZones],
            maxZoneReached: progress.maxZoneReached,
            zoneStats: progress.zoneStats,
            timestamp: Date.now(),
            version: CONFIG.VERSION,
        }));
    }

    load() {
        const raw = localStorage.getItem(this.STORAGE_KEY);
        return raw ? JSON.parse(raw) : { unlockedZones: [0], maxZoneReached: 0 };
    }
}
```

## 六、通关条件设计建议

| 关卡 | 建议条件 | 难度递进 |
|------|----------|----------|
| 1→2 | 吃到3个绿色宝珠 | 教程关 |
| 2→3 | 身长达到8节 | 学习成长 |
| 3→4 | 消灭1个敌人 | 引入战斗 |
| 4→5 | 2分钟内存活 | 生存挑战 |
| 5→6 | 吃到1个橙色宝珠 | 引入高级食物 |
| ... | 递增难度 | 每层引入新机制 |

**每5关一个难度档**：
- **第1~5关（基础层）**：学习移动、吃宝珠、躲避障碍
- **第6~10关（战斗层）**：引入敌人，学习战斗
- **第11~15关（策略层）**：宝珠选择、路径规划、资源管理
- **第16~20关（技巧层）**：狭窄通道、移动障碍、限时挑战
- **第21~25关（终极层）**：综合考验，最终Boss

## 七、自由回访技术实现

**方案A：懒加载区域**（推荐）
- 所有 25 个区域的配置数据预先定义（障碍物位置、敌人类型、宝珠分布）
- 实体按需生成：进入区域时 spawn，离开时冻结/销毁
- 优点：内存友好
- 缺点：跨区域时有轻微卡顿

**方案B：全实体保留 + 活跃度分级**
- 近距离区域（当前+相邻4格）：完整 update
- 中距离（2~3格外）：只保留位置，不 update
- 远距离（4格外）：序列化存储，实体清空
- 优点：回访无缝
- 缺点：管理复杂

**推荐方案A**，因为 25 个区域中同时活跃的只有 1~3 个，没必要全保留。

## 八、主要技术风险

| 风险 | 严重度 | 说明 |
|------|--------|------|
| **架构改动量大** | 🔴高 | game.js 已经 2300+ 行，关卡系统会引入大量新逻辑 |
| **实体生命周期管理** | 🟡中 | 宝珠/敌人需要从"全局"改为"区域作用域"，涉及碰撞检测、空间网格等 |
| **重生机制** | 🟡中 | 死亡后在哪里重生？当前区域起点？最近检查点？ |
| **幼体系统冲突** | 🟡中 | 幼体会跟着玩家跨区域，需要处理区域跟随逻辑 |
| **子弹跨区域** | 🟢低 | 子弹飞出当前区域后需要销毁或转交相邻区域 |
| **断尾/尸体系统** | 🟢低 | 尸体沉底在当前区域，回访时需保留 |

## 九、与现有系统的冲突评估

### 🔴 饥饿系统
- 当前：虫虫持续消耗身体，需要不断吃宝珠
- 问题：如果关卡1的宝珠吃完了，门还没打开，虫虫会饿死
- **建议**：关卡切换时重置饥饿计时器，或每个关卡内保证宝珠充足

### 🔴 敌人系统
- 当前：敌人从地图边缘随机生成，全图追逐
- 问题：玩家在关卡1打怪，敌人可能从关卡3穿墙过来
- **建议**：敌人锁定在当前关卡区域，不跨区追逐

### 🟡 幼体/后代系统
- 幼体会跟随玩家，但跨区时可能被门卡住
- **建议**：幼体跟随玩家自动穿越已解锁区域，未解锁区域的门对幼体同样阻挡

### 🟡 子弹系统
- 子弹飞到区域边界时如何处理？
- **建议**：子弹到达区域边界时自动消失（合理——子弹打到墙上）

### 🟢 相机系统
- 当前的 lerp 跟随天然适配区域切换，不需要大改

### 🟢 宝珠系统
- 只需把生成范围从"全图"改为"当前区域+相邻区域"

## 十一、建议方案

> **不要做成纯线性关卡**，而是做成"开放世界 + 锁钥结构"。

1. 地图保持 4000×3000 的整体开放空间
2. 在 Z 字形路径的关键节点放置**障碍门**（视觉上是藤蔓/岩石/能量墙）
3. 门的条件透明（靠近时显示"需要身长15节"等提示）
4. **玩家可以自由探索已解锁的所有区域**，不是被困在一个 800×600 的小格子里
5. 通关后回访有隐藏内容（收集品/新敌人/彩蛋）

### 渐进式实现路线

1. **Phase 1**：ZoneManager + 区域数据（纯数据层，不改现有逻辑）
2. **Phase 2**：Barrier 门系统（先做最简单的"墙"阻挡）
3. **Phase 3**：宝珠/敌人按区域生成（改造 Food.random 和敌人生成）
4. **Phase 4**：通关条件 + 进度存储
5. **Phase 5**：自由回访 + 区域实体懒加载

这样每一步都可以独立测试，不会一次性搞崩游戏。
