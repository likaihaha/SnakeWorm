# SnakeWorm 游戏开发技能

## 描述

用于管理和开发 SnakeWorm 2D 蚯蚓游戏的技能。当用户提到"改参数"、"改设计"、"更新文档"、"版本迭代"、"新增功能"、"修改功能"时触发此技能。

## 触发词

改参数、改设计、更新文档、版本迭代、新增功能、修改功能、调整参数、修改设计、同步文档、SnakeWorm、蚯蚓游戏

## 核心规则

### 文档同步规则

**当用户要求修改游戏参数或设计时，必须同步更新设计文档：**

1. **修改现有参数/设计**：
   - 在 `SnakeWormDesign_v0.3.md` 原文档中直接修改
   - 更新相关章节（配置参数、机制说明等）
   - 在文档末尾的"开发日志"中记录修改内容

2. **新增功能**：
   - 创建新版本文档：`SnakeWormDesign_v0.X.md`（X 为新的次版本号）
   - 基于当前最新版本文档复制并修改
   - 在文档头部更新"版本信息"
   - 在文档末尾的"版本历史"中新增版本记录
   - 将 `SnakeWormDesign_v0.3.md` 标记为"已归档"

3. **修改旧功能**：
   - 创建新版本文档：`SnakeWormDesign_v0.X.md`
   - 基于当前最新版本文档复制并修改
   - 在文档头部更新"版本信息"
   - 在文档末尾的"版本历史"中新增版本记录
   - 将 `SnakeWormDesign_v0.3.md` 标记为"已归档"

### 版本号规则

- **主版本号**：重大架构变更时递增（如 v1.0 → v2.0）
- **次版本号**：新增功能或修改现有功能时递增（如 v0.3 → v0.4）
- **实验版本**：分支实验使用 `-实验版` 后缀（如 v0.3-实验版）

## 工作流程

### 1. 参数修改流程

```
用户请求修改参数 → 定位相关配置 → 修改代码 → 同步更新文档 → 记录开发日志
```

**示例**：
```
用户："把断尾无敌时间从 13 秒改为 9 秒"
AI：
  1. 修改 CONFIG.SPLIT_INVINCIBLE_TIME: 13.0 → 9.0
  2. 更新 SnakeWormDesign_v0.3.md 的"配置参数"章节
  3. 在"开发日志"中记录修改
```

### 2. 新增功能流程

```
用户请求新增功能 → 评估影响范围 → 创建新版本文档 → 实现功能 → 更新文档
```

**示例**：
```
用户："新增道具系统"
AI：
  1. 创建 SnakeWormDesign_v0.4.md（基于 v0.3）
  2. 在 v0.4 文档中新增"道具系统"章节
  3. 将 v0.3.md 标记为"已归档"
  4. 实现功能代码
```

### 3. 修改旧功能流程

```
用户请求修改功能 → 评估影响范围 → 创建新版本文档 → 修改功能 → 更新文档
```

**示例**：
```
用户："修改咬尾分裂机制，改为咬头死亡"
AI：
  1. 创建 SnakeWormDesign_v0.4.md（基于 v0.3）
  2. 在 v0.4 文档中修改"咬尾分裂"章节为"咬头死亡"
  3. 将 v0.3.md 标记为"已归档"
  4. 修改功能代码
```

## 注意事项

1. **文档同步是强制性的**：任何代码修改都必须在文档中体现
2. **版本号递增**：新增/修改功能时，版本号必须递增
3. **版本归档**：新版本创建后，旧版本标记为"已归档"
4. **开发日志**：每次修改都要在"开发日志"中记录
5. **版本历史**：维护完整的版本演进历史
