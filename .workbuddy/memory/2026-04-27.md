## v0.63: 修复iframe嵌入比例被压扁
- 问题：上传到比赛网页后，游戏画面比例被压扁（宽度撑满、高度被压缩）
- 根因：CSS `max-width: 100%` + `max-height: 65vh` 与 JS resizeCanvas() 的等比缩放冲突；resizeCanvas() 用 window.innerWidth/Height 在 iframe 环境下不准确
- 修复：
  1. 移除 CSS 中所有 max-width/max-height 限制（基础样式和横屏媒体查询）
  2. resizeCanvas() 改用 canvas.parentElement.clientWidth/Height 获取可用空间
  3. JS 的 Math.min(scaleX, scaleY) 保证始终等比缩放
- 已提交推送

## v0.64: 修复浏览器缩放后canvas不更新
- 问题：浏览器缩放（Ctrl+/Ctrl+-）后，canvas尺寸不更新，画面异常
- 根因：浏览器缩放不会触发 window.resize 事件
- 修复：添加三重检测机制
  1. 监听 wheel 事件（Ctrl+滚轮缩放）
  2. 定期检测 devicePixelRatio 变化（Ctrl+加减键、菜单缩放）
  3. ResizeObserver 监听容器尺寸变化（最可靠）
- 已提交推送

## v0.66: 修复canvas尺寸过小
- 问题：canvas 变得非常小，游戏画面缩成一团
- 根因：canvas.parentElement 是 position:relative 的 div，没有设置宽度，会收缩适应 canvas，导致循环依赖
- 修复：
  1. 给外层 flex 容器添加 id="gameContainer"
  2. resizeCanvas() 优先使用 #gameContainer 的尺寸
  3. fallback 到 document.body，最后用 window
- 已提交推送

## v0.67: 修复canvas尺寸不稳定
- 问题：canvas 尺寸剧烈波动（一会巨大一会极小）
- 根因：
  1. ResizeObserver 监听 canvas.parentElement，resizeCanvas 改变 canvas 尺寸 → 父元素尺寸变化 → 触发 Observer → 再次 resize，形成反馈循环
  2. window.innerWidth/Height 在缩放时变化剧烈
- 修复：
  1. 移除 ResizeObserver（避免反馈循环）
  2. 改用 documentElement.clientWidth/Height（更稳定，不包括滚动条）
  3. 添加最小缩放比例 0.35（防止 canvas 过小）
  4. 恢复 CSS max-width: 100% 和 height: auto 作为兜底保护
  5. resize 事件添加防抖（100ms）
- 已提交推送

## v0.68: 修复出场动画后卡在waitingForPlayer
- 问题：PC端(Edge/Chrome)进入游戏后，出场动画完成后游戏卡住不动，需要手动移动鼠标到白圈才能开始
- 根因：出场动画phase 2检测到distanceToMouse < 5（head和enterTargetPos都在中心），直接完成动画(isEntering=false, enterPhase=3)，但waitingForPlayer仍为true，游戏永远卡在等待状态
- 修复：在update()的waitingForPlayer分支中，出场动画完成后自动清除waitingForPlayer状态，游戏自动开始
- 移除了debug用的console.log语句
- 已提交推送，标签v0.68

## v0.69: 修复PC端误判为触摸设备
- 问题：PC端(Edge/Chrome)进入游戏后显示虚拟摇杆和射击按钮，干扰操作
- 根因：`navigator.maxTouchPoints > 0` 在Windows PC上经常误报（某些驱动报告有触摸能力），导致isTouchDevice误判为true
- 修复：
  1. 移除基于能力检测的触摸设备判断（`'ontouchstart' in window || navigator.maxTouchPoints > 0`）
  2. 改为实际触摸事件触发后才启用移动端UI
  3. 首次touchstart时动态初始化摇杆、射击按钮、全屏按钮等移动端UI
  4. 同时还原了v0.68的自动开始游戏改动（用户不需要）
- 已提交推送，标签v0.69

- 修复：

  2. 玩家碰撞检测：添加 `player.isAlive` 检查
  3. 玩家预警/撞墙检测：添加 `player && player.isAlive` 检查
  4. 咬尾分裂检测：添加 `player` 存在性检查
  5. 磁力结算：重置连击前再次检查 `player` 存在性
